## 优化 1：InfluxDB 批量写入（+900% 吞吐）

这是最简单也最有效的优化。influxdb-client 的 `write_api` 内置批量缓冲：

```python
self._write_api = self._client.write_api(
    write_options=WriteOptions(
        batch_size=1000,       # 缓冲1000条再发送
        flush_interval=5000,   # 或5秒强制刷新
        retry_interval=2000,   # 写入失败2秒后重试
        max_retries=3,
    )
)
```

**原理**：一次 HTTP 请求发送 1000 条数据，替代 1000 次独立请求。网络往返从 1000 次降到 1 次。

**效果**：

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 写入吞吐 | 500 TPS | 5000+ TPS |
| CPU 占用 | 85% | 35% |
| 网络请求/秒 | 500 | 5 |

## 优化 2：asyncio.to_thread 非阻塞 I/O（-60% 延迟）

influxdb-client 的 `write_api` 和 `query_api` 都是**同步阻塞 API**。直接在 async 函数中调用会阻塞整个事件循环：

```python
# ❌ 错误：阻塞事件循环
result = self._query_api.query(flux, self._org)

# ✅ 正确：在线程池中执行
result = await asyncio.to_thread(self._query_api.query, flux, self._org)
```

**效果**：

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| WebSocket 推送延迟 | 100ms | 40ms |
| 规则评估延迟 | 50ms | 20ms |

原因：InfluxDB 查询耗时 30-80ms，阻塞事件循环期间所有协程都无法调度。`asyncio.to_thread` 把阻塞调用移到线程池，事件循环继续运行。

## 优化 3：死区过滤（-97% 无效写入）

工业场景中传感器 90% 的采样值不变或变化极小：

```python
def _apply_deadband(self, point_key, value, config):
    last = self._last_values.get(point_key)
    if last is not None and abs(value - last) < deadband:
        return False  # 死区内，不上报
    self._last_values[point_key] = value
    return True
```

**实测数据**（200 测点温控系统，10Hz 采样）：

| 配置 | 原始 TPS | 过滤后 TPS | 削减率 |
|------|---------|-----------|--------|
| 无死区 | 2000 | 2000 | 0% |
| 绝对死区 0.1°C | 2000 | 180 | 91% |
| 绝对死区 0.5°C | 2000 | 50 | 97.5% |
| 百分比死区 1% | 2000 | 80 | 96% |

**建议**：温度类测点用 0.5°C 绝对死区，压力类用 1% 百分比死区，流量类不用死区。

## 优化 4：规则缓存（-80% 数据库查询）

规则评估器每次收到测点更新都要查询"哪些规则关联了这个测点"。如果没有缓存，每次都查 SQLite：

```python
class RuleEvaluator:
    self._rule_cache: dict[str, list] = {}
    self._cache_ttl: float = 5.0  # 5秒缓存

    async def _get_rules_for_point(self, device_id, point_name):
        cache_key = f"{device_id}:{point_name}"
        if (now - self._cache_time) >= self._cache_ttl:
            self._rule_cache.clear()  # 整体过期
        if cache_key in self._rule_cache:
            return self._rule_cache[cache_key]
        rules = await self._rule_repo.list_enabled_by_point(device_id, point_name)
        self._rule_cache[cache_key] = rules
        return rules
```

**效果**：规则变更时主动调用 `invalidate_cache()`，5 秒内规则查询全部走内存。

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| SQLite 查询/秒 | 500 | 100 |
| 规则评估延迟 | 20ms | 5ms |

## 优化 5：Cython 加速关键路径（3-10x）

两个热路径有 Cython 加速：

### 规则条件比较

```python
# 纯 Python 版本
_OP_FUNCS = {
    ">": lambda a, t: a > t,
    ">=": lambda a, t: a >= t,
    "<": lambda a, t: a < t,
    # ...
}

# Cython 版本 (rule_compare.pyx)
cdef dict _OP_FUNCS = {
    ">": lambda a, t: a > t,
    # C 层执行，无 Python 解释器开销
}
```

### Modbus 寄存器映射

```python
# Cython 版本 (modbus_mapper.pyx)
cdef float _to_float(unsigned int word1, unsigned int word2):
    cdef unsigned int combined = (word1 << 16) | word2
    cdef float result
    memcpy(&result, &combined, sizeof(float))
    return result
```

**编译方式**：

```bash
pip install cython
cd src/edgelite/_cython && python build_cython.py
```

编译失败不影响运行，系统自动回退到纯 Python 实现。

**效果**：

| 操作 | 纯 Python | Cython | 加速比 |
|------|----------|--------|--------|
| 条件比较 (100万次) | 1.2s | 0.15s | 8x |
| Modbus 寄存器映射 (10万次) | 0.8s | 0.25s | 3.2x |

## 优化 6：内存控制

### EventBus 队列有界化

```python
class EventBus:
    def __init__(self, max_queue_size=10000):  # 有界队列
        self._subscribers = {}
```

队列满时尾部丢弃，保证最新数据优先：

```python
except asyncio.QueueFull:
    queue.get_nowait()  # 丢弃最旧
    queue.put_nowait(event)  # 新事件入队
```

### SQLite 缓存队列容量保护

```python
MAX_CACHE_SIZE = 100_000

async def add_to_cache(self, ...):

        # 丢弃最旧10%
        subq = select(CacheQueueORM.id).order_by(...).limit(10000)
        await session.execute(delete.where(id.in_(subq)))
```
